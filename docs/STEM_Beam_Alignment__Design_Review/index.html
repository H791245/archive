STEM Beam Alignment — Design Review

Structured summary of the discussion about beam shift, alignment logic, and storage strategy.
Agenda
Context
–
  • Clarify which beam/image shift values are used and stored (user vs system vs corrector).
  • Understand the alignment flow: acquire images → measure pixel shift → compute beam correction.
  • Decide where to persist shifts: LG file, STEM FEC registers, or exported elements.
  • Review why we zero some elements and what risks that introduces.
  • Define microscope experiments to validate assumptions and diagrams.
Key concepts
What’s being discussed
–
  • beamAlignmentShift: corrective shift applied after comparing images.
  • Difference between image shift, user‑induced shift, and system/corrector shift.
  • Zeroing of exported elements before running alignment to avoid user interference.
  • Three alignment attempts with an acceptance window for the measured shift.
  • Goal: keep FEC registers independent from corrector state while still compensating drift.
Decisions
Current stance
–
  • Verify where beamAlignmentShift actually lives (LG file vs FEC vs model) before changing logic.
  • Do not change corrector‑related elements for now; treat them as stable references.
  • Use the beam shift that has no dependency on exported elements as the main stored reference.
  • Treat the current zeroing logic as suspicious; validate on a real system before refactoring.
  • Keep existing behavior until microscope tests confirm the intended data flow.
Open questions
Unknowns
–
  • Is beamAlignmentShift stored in the LG file, a fixed STEM register, or an internal model?
  • Does Alam‑STEM use a dedicated beam shift or reuse the generic one?
  • Who “owns” the effective shift: user actions, alignment code, or corrector configuration?
  • Should shifts be stored per mode (STEM1/2/3, voltages, currents) or shared globally?
  • Why do we need to re‑zero if the corrector is supposed to be stable?
Action items
What happens next
–
Code & registers
  • Trace beamAlignmentShift through the code to its storage location.
  • Compare the current diagram to real implementation and fix mismatches.
  • Log before/after values of beam shift, image shift, and related registers with no code changes.
Microscope tests
  • Switch beam currents (e.g. 100 pA ↔ 500 pA) and see if shift changes or only intensity/focus.
  • Switch accelerating voltage (e.g. 25 kV → 5 kV) and confirm shifts vs pure focus effects.
  • Probe Alam‑STEM vs STEM behavior to see whether they share or separate beam shifts.
Design & alignment strategy
  • Write a short design note explaining current logic, risks, and the desired target model.
  • Align with domain experts on who is allowed to modify which registers or files.
  • Create work items for a safer alignment refactor if tests justify it.
